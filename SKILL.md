# Agentic Engineering TDD (RED - GREEN - REVIEW)

## 개요

Agentic Engineering TDD는 Coding Agent가 코드를 작성하기 전에 **사람이 먼저 계획을 검토**하고,
구현이 끝난 뒤에는 **사람이 결과를 리뷰**하는 human-in-the-loop 사이클이다.

기존 TDD의 Red-Green-Refactor와 다른 점: Agent가 자율적으로 다음 테스트를 계속 진행하지 않는다.
매 사이클마다 "계획 승인 → 구현 → 결과 승인"의 관문(gate)을 사람이 통과시켜야 다음 단계로 넘어간다.

핵심 원칙: **사람의 승인 없이는 다음 단계로 넘어가지 않는다.**

## 사이클 구조 (RED는 두 개의 승인 관문으로 나뉜다)

한 사이클은 **네 번의 정지점**을 거친다. 각 정지점마다 사람의 명시적 승인(또는 수정 요청)을
받기 전까지 다음으로 넘어가지 않는다: (1) Plan 승인 → (2) Test Code 승인 → (3) GREEN 구현
후 리뷰 승인 → (4) 커밋. 여러 사이클(예: 카타의 테스트 케이스 여러 개)을 진행할 때도
사이클마다 이 네 정지점을 반복하며, 한 번의 승인이 다음 사이클에 대한 승인을 의미하지 않는다.

### 1a. RED — 목표 설정 + Plan.md 작성 (1차 승인 관문)

- 이번 사이클에서 달성할 목표(goal)를 정확히 정의한다 (모호하면 사람에게 먼저 질문한다).
- 목표, 구현 예정 테스트 케이스, 구현 접근 방식, 파일 위치, 커밋 계획을 담은 `Plan.md`를 작성한다.
  여러 사이클을 한 번에 계획할 경우 하나의 `Plan.md`에 테스트 케이스 목록으로 묶어도 되지만,
  각 사이클을 실제로 진행할 때는 아래 1b 단계에서 반드시 별도로 재확인·재승인을 받는다.
- **Plan.md 작성 후 코드를 작성하지 않고 멈춘다.** 사람에게 검토를 요청한다.
- 사람이 승인하면: `Plan.md`만 커밋한다 (RED 커밋). 코드는 아직 없다.
- 사람이 수정을 요청하면: Plan.md를 수정하고 다시 검토를 요청한다. 승인 전까지 반복한다.

### 1b. RED — Test Code 변환 (2차 승인 관문)

- 승인된 Plan(또는 Plan 안의 해당 테스트 케이스)을 실제 테스트 코드로 변환해 작성한다.
- **이 시점에도 아직 구현 코드(프로덕션 코드)는 작성하지 않는다.** 테스트 코드만 작성한다.
- 작성한 테스트 코드를 사람에게 제시하고, Plan의 의도에 맞게 작성되었는지 확인을 요청한다.
  - 승인 전에는 GREEN으로 넘어가지 않는다.
  - 수정 요청이 오면 테스트 코드를 고치고 다시 확인을 요청한다.
- **특수 케이스 — GREEN 없이 이미 통과하는 테스트**: 기존 구현만으로 새 테스트가 이미 통과하는
  경우가 있다 (예: 일반화된 로직이 새 케이스도 우연이 아니라 실제로 커버하는 경우). 이때는:
  1. GREEN 구현을 건너뛴다.
  2. 왜 통과하는지 설명한다 (어떤 기존 로직이 이 케이스를 커버하는지).
  3. 이 테스트가 우연히 통과한 것이 아니라 실제로 유효한 검증(회귀 테스트)인지 사람에게 확인받는다.
  4. 확인되면 GREEN 단계 없이 REVIEW/커밋 단계로 넘어간다 (RED → REVIEW).

### 2. GREEN — Plan대로 최소 구현

- **승인된 Plan.md 범위 안에서만** 구현한다. Plan에 없는 기능·리팩토링·최적화를 임의로 추가하지 않는다.
- 테스트를 통과시키는 최소한의 코드만 작성한 뒤, 실제로 `pytest`를 실행해 통과하는지 직접 확인한다.
- 이 단계에서는 **아직 커밋하지 않는다.** GREEN 코드는 사람 리뷰 전까지 커밋 대기 상태로 둔다.
- 구현 중 Plan의 범위를 벗어나야 할 필요를 발견하면(예: 엣지 케이스 누락), 코드를 계속 진행하지 말고
  사람에게 알리고 Plan.md를 먼저 수정한 뒤 승인받는다 (1a로 되돌아감).

### 3. REVIEW — 사람 리뷰 후 커밋

- 구현된 코드와 테스트 결과를 사람에게 제시한다.
- 리뷰 시 확인할 것:
  - Plan.md에 명시된 것 **이상**으로 구현된 것이 있는가? (범위 이탈)
  - Plan.md에 명시된 것 중 빠진 것이 있는가?
  - 리팩토링이 필요한 부분이 있는가? (예: 반복되는 원시 조건문을 의미 있는 이름의 헬퍼 함수로 추출)
- 사람이 리팩토링을 요청하면 반영하고, `pytest`로 여전히 전체 통과하는지 확인한 뒤 다시 리뷰를 요청한다.
- 사람이 승인하면: 코드(구현 + 테스트, GREEN을 건너뛴 경우 테스트만)를 커밋한다.
- **사람 승인 없이 코드를 커밋하지 않는다.**

## 커밋 규칙

| 시점 | 커밋 대상 | 조건 |
|---|---|---|
| RED | `Plan.md` | 사람이 Plan을 승인한 후 |
| GREEN | 구현 코드 + 테스트 | 사람이 REVIEW에서 승인한 후 |
| REVIEW (GREEN 생략) | 테스트 코드만 | 기존 구현으로 이미 통과함을 사람이 확인한 후 |

원격 저장소가 연결되어 있다면 승인된 커밋은 바로 `git push`해서 기록을 남긴다.
사이클마다 하나의 커밋으로 남기고, 커밋 메시지에 단계와 사이클 번호를 명시한다.
예: `RED 1: Plan.md for bowling score cycle`, `GREEN 2: sum all rolls`,
`GREEN 3: add spare bonus (refactored per review)`, `REVIEW 5: already covered, no GREEN needed`.

## Iron Law (철의 법칙)

1. **Plan.md가 사람에게 승인되기 전에는 테스트 코드도, 구현 코드도 작성하지 않는다.**
2. **테스트 코드가 사람에게 승인되기 전에는 구현(GREEN) 코드를 작성하지 않는다.**
3. **사람이 GREEN(또는 스킵된 GREEN) 결과를 리뷰하고 승인하기 전에는 커밋하지 않는다.**
4. Plan에 없는 범위는 그 자리에서 임의로 구현하지 말고, 먼저 Plan.md를 갱신해 재승인받는다.
5. 여러 사이클을 진행할 때, 한 사이클에 대한 승인이 다음 사이클에 대한 자동 승인이 되지 않는다.
   사이클마다 Plan/Test Code/REVIEW 승인을 각각 새로 받는다.

이 중 하나라도 건너뛰었다면 Agentic Engineering TDD가 아니라 그냥 TDD(또는 그 이하)를 한 것이다.

## 흔한 위반 패턴 (Red Flags)

- Plan.md 승인 답변을 받지 않고 바로 구현으로 넘어감
- 테스트 코드를 사람에게 보여주지 않고(혹은 승인받지 않고) 바로 GREEN 구현으로 넘어감
- Plan.md에 없는 헬퍼/최적화/추가 테스트 케이스를 "겸사겸사" 구현
- GREEN 구현 후 사람 리뷰를 요청하지 않고 바로 커밋
- 리뷰에서 지적받은 리팩토링을 반영하지 않고 그대로 커밋
- 여러 사이클을 사람 승인 한 번으로 뭉뚱그려 한꺼번에 진행 ("나머지도 알아서 해줘"를 문자 그대로
  승인 생략으로 해석)
- RED/GREEN 커밋을 하나로 합쳐서 커밋 (단계 구분이 사라짐)
- 테스트가 GREEN 없이 이미 통과했는데 그 이유를 설명하지 않고 그냥 다음으로 넘어감

## 검증 체크리스트

사이클을 완료로 표시하기 전에:

- [ ] Plan.md에 목표/테스트 케이스/구현 접근/파일 위치/커밋 계획이 모두 있다
- [ ] Plan.md에 대해 사람의 명시적 승인을 받았다
- [ ] RED 커밋(Plan.md만)이 별도로 존재한다
- [ ] 테스트 코드를 사람에게 제시하고 Plan에 맞는지 승인을 받았다 (구현 전에)
- [ ] 구현이 Plan.md 범위를 벗어나지 않았다
- [ ] `pytest`로 테스트 통과를 직접 확인했다 (또는 GREEN 없이 통과한 이유와 유효성을 설명했다)
- [ ] GREEN 코드에 대해 사람의 리뷰(범위 이탈/리팩토링 필요성)를 받았다
- [ ] 리뷰에서 나온 수정 요청을 반영했다 (있었다면)
- [ ] 사람의 승인 후에만 커밋했다

체크박스를 모두 채울 수 없다면, 그 단계로 돌아가 사람의 승인을 받아야 한다.
